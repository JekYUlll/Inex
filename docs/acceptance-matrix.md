# Inex Verification and Acceptance Matrix

This matrix is the binding evidence checklist for the pre-alpha release gate.
Test names may change, but no row can be waived without updating the threat
model and product status.

| Area | Required cases | Passing evidence |
|------|----------------|------------------|
| Envelope parsing | Bit flips in magic/version/flags/length/header/body/tag; every truncation boundary; appended bytes | All fail closed and return no plaintext. |
| Deterministic CBOR | Duplicate/out-of-order keys, non-minimal integer/length, indefinite values, unknown fields/features | All rejected before allocation/decryption beyond limits. |
| AAD and path binding | Change nonce, suite, KDF id, file/vault id, epoch, logical path, timestamps, flags, feature list, or draft base etag | Authentication or semantic validation fails; an OS rename is reported as path mismatch. |
| Nonce behavior | Repeated save of identical bytes; rename; encrypted draft | Fresh RNG nonce every time and distinct envelopes. |
| Password slots | Wrong password/slot; changed KDF/wrap data; changed other slot/features/MAC; add/remove/change password | Invalid states fail uniformly; all EDRY hashes remain unchanged for slot-only operations. |
| KDF resources | Synthetic fast/slow/discrete/noisy calibration covering all five outcome branches; absent/explicit RPC KDF; below/above creation cap; valid older weak vault; stronger authenticated slot rewrap | Deterministic injected tests are authoritative for bounded-search semantics. Default creation selects ops 3–20 at fixed 64 MiB without timing user secrets; explicit RPC memory outside exact 64 MiB fails before root creation; reader ceiling remains broader; add/change never lowers either authenticated work-factor component. |
| KDF diagnostic and native evidence | Exact no-argument CLI parsing before password/query setup; strict 20-line ASCII schema/order; calibration failure before report emission; isolated product-state residue; separate packaged CLI/daemon runtime probes plus three fresh calibration-attempt processes on native Linux x64/arm64 and Windows x64/arm64 MSVC | Each valid report binds `inex-kdf-calibration-v1`, runtime identity, fixed policy, selected observation/count, and exactly one of the five outcomes. The external canonical JSON retains attempts 1–3 with no retry/selection and binds clean source, artifact, harness, runtime, host, and resource observations. It validates emitted selected evidence only; deterministic tests remain the search-semantics authority. Wine, cross, and emulation do not satisfy native rows. The current harness accepts Linux only and fails closed on Windows until the ADS and suspended-Job boundaries are implemented. |
| Secure buffers | Init failure, mlock unavailable, auth failure, panic/error paths, lock/drop | Keys and owned plaintext buffers are wiped; degraded memory-lock state is reported without secrets. |
| Atomic file write | Inject failure at staging create/write/flush/sync, etag recheck, namespace move, post-error state check, and durability barrier | Old or new target remains a complete authenticated envelope; ambiguous namespace state is explicit; no partial plaintext/target. |
| Atomic metadata write | Same fault points during password-slot transaction | At least one unlockable authenticated metadata copy remains; EDRY files unchanged. |
| Concurrency | Two independent sidecars write the same etag; rename/write; delete/write | Exactly one mutation commits; the other receives a stable conflict with current etag. |
| Logical paths | Traversal, absolute/backslash, controls, reserved names/chars/devices, leading/trailing spaces, DOS `~digit` aliases, NFC/case collisions, coexisting aliases, 251-byte filename and 1024-byte path boundaries | Same accept/reject result on Linux and Windows; no vault escape or Git/NTFS alias; Windows native Git round-trip uses `core.longPaths=true`. |
| Filesystem boundary | Network/FUSE root, nested mount/bind mount, symlink, Windows junction/reparse, hardlink and identity swap | Unsupported storage fails before KDF/read/write/recovery; no external path is opened, retired, or deleted. |
| Text byte round trip | Chinese, emoji, combining characters, BOM, LF/CRLF/mixed newline, empty and max-size input | Decrypted bytes exactly equal input; no implicit normalization/newline rewrite. |
| RPC framing | Partial/coalesced frames, invalid length, oversize/deep JSON, batch/notification, stdout noise | Parser stays synchronized or terminates safely; no allocation beyond limits. |
| RPC leakage | Canary password/content/token through success, all errors, malformed input and crash | Canary absent from argv, environment, stderr, diagnostics, backup names, and logs. |
| Session lifecycle | Wrong token, explicit lock, stdin EOF, editor crash, idle expiry, daemon shutdown | Old capabilities fail uniformly; index/cache/key access is gone. |
| Search freshness | External replacement, same-size in-place tamper, preserved timestamps, second sidecar save | Every query verifies current ciphertext etags; changed/corrupt storage invalidates the plaintext index before returning hits. |
| Opaque assets | Feature negotiation, 64 MiB boundary, path/type collision, tamper, sequential chunking, open/close/lock/session loss | Feature-1 vaults classify every `.asset.enc`; authentication precedes plaintext return; one bounded zeroizing daemon buffer exists per session and is destroyed on every terminal path. |
| Controlled image preview | Relative PNG/JPEG/WebP, traversal/encoded separator/external URL, pixel bomb, animation, fenced code/raw HTML, edit/hide/lock races | Only authenticated same-vault bounded images become revocable `blob:` URLs; stale transfers are rejected and no plaintext file or ordinary `TextDocument` is created. |
| Encrypted drafts | Custom editor dirty backup, restore before unlock, stale base etag, corrupted backup | Backup disk bytes are EDRY draft ciphertext only; restore authenticates and never overwrites stale base silently. |
| VS Code residue | Edit/undo/save/dirty close/crash/restore with Hot Exit and Local History combinations | No plaintext canary in workspace storage, backups, local history, logs, telemetry, or extension state. |
| Sublime residue | Edit/save/close/crash/session restore/recent-files matrix | No plaintext canary on disk before the client leaves experimental status. |
| Locked Git driver | Normal Git merge invokes driver without an unlock broker | `%A` hash unchanged, nonzero status, stages 1/2/3 retained, no plaintext artifact. |
| Unlocked Git resolution | Clean/conflicting diff3, delete/modify, rename/modify, Unicode/space path | Clean result is valid EDRY; unresolved markers are encrypted and conflict flag authenticates. |
| Driver installation | Fresh clone before/after `inex git install-driver` | `.gitattributes` travels with Git; local `.git/config` is installed only by explicit command. |
| Copy import | Symlink/junction, overlap, path collision, source changes, disk fault, existing destination | Source hashes/bytes remain unchanged; destination is absent or a clearly marked ciphertext staging vault. |
| Repository snapshot import | Clean/dirty SHA-1 HEAD, index/worktree/control mutation, forbidden raw `FSMN` extension, filters/LFS/submodule/link/hardlink, Markdown+25,074,521-byte asset, resource boundaries, target-object plaintext scan, every construction/durability/publication kill edge, retry/reconcile, Linux and native Windows | Source history and worktree remain byte/identity/semantic identical; target is absent, a retained explicit candidate, indeterminate, or one complete new parentless ciphertext repository with stable terminal fields. `FSMN` is rejected rather than treated as an ignorable cache extension or ordinary entry-flags bit. Linux trusted-local functional coverage alone is not GA evidence. |
| Repository publication claim v2 (GA) | Literal `INEXPUB\0`; exact big-endian offsets/version/scheme/total/lengths; 16-byte nonzero id; Linux dev/inode scheme 1, Windows all-modern FileId128 scheme 2, and all-legacy file-index scheme 3; one scheme for all four marker identities and every seal identity in sections 1/2/8/9; exact marker-parent binding, domain, portable names, 998-byte maximum/1,024-byte ceiling, and trailing SHA-256; exact staging grammar and reserved destination prefix; legacy/v2 paths and private reserved set | Only exact canonical v2 at `.vault-local/import-publish-marker-v2` enters recovery. Clean private state is exact bound `mutation.lock`; marker-aware state adds only v2. Any mixed identity scheme, modern/fallback drift, wrong offsets/limits/domain/names/externally expected id/identity/digest, legacy+v2, case aliases, unknown reserved entries, copied/rebound markers, encoded identity drift, and malformed bytes preserve state and fail closed. Same mount instance is an operational precondition, not a wire proof: identity-preserving remount is undetectable. Without an external expected id, fresh recovery proves internal id/seal/digest and one-claim binding rather than an independent id oracle. |
| Repository candidate seal v1 (GA) | Golden and independently regenerated `INEXCS1\0` stream with exact header, nine ordered counted/length-framed sections and terminator; physical manifest, worktree allowlist, SHA-1 HEAD/ref, raw index `CE_NAMEMASK` pathname length, separately decoded `CE_STAGEMASK` stage, normalized-zero remaining flags, forbidden `FSMN`, tree/commit/object semantics, exact section-8 role/path/kind control map, and exact private baseline; marker-free durable audit followed by held existing lock, create-new empty marker, held marker identity, uniform-scheme proof, streaming seal, canonical marker sync, seal recomputation/audit, then move; pathname-only marker skipping negatives; independent 1,000,000-record section-1/7/8 limits, section-1 root count, section-5 directory bound, 100,003 worktree/index, 1,034-byte section-1 physical path, 1,024-byte semantic paths, per-path-bearing-section 256 MiB sums, 68 MiB file/object, checked arithmetic, and streaming boundaries | Seal is exactly 32-byte SHA-256. The marker-free baseline binds section-9 lock identity before lock acquisition; acquiring the lock does not alter its identity/body, and the later marker seal re-proves the same held identity without a dependency cycle. Initial and recovery traversal open and hold the exact marker and exclude its exact pathname only when the encountered identity equals the held identity; missing/replaced/lookalike markers fail and pathname alone is never skipped. The fresh target-only audit needs no process-local `TargetRepository`, source Git/body, password, KDF, unlock, or decryption. Sections 1, 2, 4, 5, 8, and 9 have independent 256 MiB path sums; section-3 ref names do not count. Extra reachable/unreachable objects and any physical or semantic drift fail. The unkeyed seal is explicitly not same-UID authentication. |
| Repository initial publisher lock (GA) | After marker-free durable baseline and before marker creation: exact existing zero-byte `mutation.lock`, identity equality with the sealed baseline, no-follow open, no-create/no-recovery API, nonblocking try-lock or bounded deadline, marker-free baseline reproof under the acquired lock, missing/replaced/busy/deadline negatives; process kill before/after marker creation and at every move/audit/sync/unlink/result edge; held-handle whole-root move on native Windows | The initial publisher holds one unchanged lock continuously across marker creation/sealing, whole-root move, post-move audit, destination/common-parent sync, marker unlink, marker-parent sync, clean audit, and terminal-result emission. It never creates or recovers the lock. The platform lock handle permits enclosing-root rename while held, including Windows delete/rename sharing. Cooperative writers cannot enter the move-to-audit-to-unlink window. Once a canonical marker is durable and the move has exposed the destination, a pre-unlink crash releases the OS lock while that marker remains for reconciliation; before the move, final remains absent and the staging marker grants no fresh-process authority. |
| Repository existing-only guard (GA) | Parameter normalization and source/destination disjointness; existing v2 dispatch before ordinary destination-exists/source Git planning/password/KDF/construction; open-existing/no-create/no-recovery lock with nonblocking try-lock or bounded deadline; missing/replaced/busy lock; pending ciphertext staging/rebind/retired state; target-derived stats; success/conflict/fresh-marker-absent terminal output; ordinary mutation attempt with canonical v2, legacy, malformed, alias, or unknown reserved publication-marker state | Guard never waits indefinitely, creates or recovers lock/vault/staging/Git, or resumes interrupted staging. Missing/busy/deadline/drift/pending/private conflicts perform zero mutation. Exact recovery opens and holds the marker and existing lock through terminal-result emission and alone may sync the published root and remove that marker. Every ordinary mutation rejects any portable-casefolded reserved marker-prefix entry before create/recovery and again under its lock; exact canonical v2 directs to repository reconcile, every other reserved conflict to manual audit. Fresh marker absence remains a hard unattributed-existing error. |
| Repository publication recovery (GA) | Real process kill before/after marker durability, directory move, destination/common-parent sync, marker open/final check/pathname removal, marker-parent sync, clean audit, and success acknowledgement; removal outcomes old-present/absent/replaced/indeterminate; fresh exact-v2, legacy, marker-free-valid, unrelated targets and interrupted staging on Linux and native Windows | Trace proves `StagingAudited -> PublishedWithMarker -> PublicationDurableWithMarker -> PublishedClean`, with the exact mutation lock continuously held and common-parent sync completed before marker removal. When the same live process proves the previously held marker identity's path absent but sync/audit is indeterminate, it enters live-only `PostUnlinkAbsentIndeterminate` and may retry only those barriers; a fresh absent path is a hard error and never infers that state. Exact v2 can reconcile; legacy, interrupted staging and marker-free fresh targets never auto-succeed or mutate. A kill after marker deletion exposes the documented acknowledgement gap unless a durable completion receipt is adopted. Cooperative final/error checks preserve replacements they observe, while the noncooperating same-UID check-to-unlink race remains an explicit non-CAS GA gate. The current random 16-byte Linux preview does not satisfy this row. |
| In-place import | Explicit confirmation absent/present and injected failures | Disabled by default; implementation cannot delete a source before verified encrypted replacement and backup. |
| Compatibility | Fixed vault/password/salt/master/slot/file id/nonce/header/body vectors | Linux and Windows produce/consume byte-identical fixtures. |
| Upgrade | Open frozen v1 fixtures in later builds; unknown major/required feature | v1 read does not rewrite; unsupported state fails closed with stable error. |
| Packaging | Offline clean build and platform package smoke test | Bundled libsodium version is reported; binaries/VSIX/Sublime package start on supported targets. |

## Filesystem residue audit roots

Release tests scan the vault, its parent staging area, OS temporary directory,
VS Code user/workspace storage and backup/local-history roots, Sublime session
and cache roots, project logs, and crash-test fixture directories for a unique
plaintext canary. CI reports paths and counts but never uploads the canary body
or a real vault.

## Release decision

- **Core pre-alpha exit:** every non-editor row through “Session lifecycle” and
  the compatibility row pass on Linux and Windows.
- **VS Code MVP:** encrypted draft and VS Code residue rows also pass.
- **Sublime supported:** Sublime residue row passes on each advertised version;
  until then the package remains experimental regardless of functional tests.
- **GA:** Git, import, upgrade, and packaging rows pass with recovery docs and
  a reproducible, offline-capable release build.
